\begin{abstract}
We present a privacy-preserving payment system for Tangle's Blueprint service protocol that enables anonymous pay-per-use access to AI inference, image generation, and general cloud compute. The system composes three layers: (1)~a cross-chain UTXO-based shielded pool~\cite{webb} with variable-amount JoinSplit transactions and multi-stablecoin wrapping, (2)~a stateless gateway contract that bridges anonymous partial withdrawals to Tangle's service lifecycle, and (3)~a prepaid credit scheme where a single zero-knowledge proof funds an account from which arbitrarily many jobs are authorized via EIP-712 signatures. Unspent funds remain shielded as change UTXOs, each withdrawal is unlinkable to prior withdrawals or the original deposit, and operators learn nothing about the payer's on-chain identity. The construction requires no modifications to Tangle's audited core contracts.
\end{abstract}

\section{Introduction}

Every major AI inference provider---OpenAI, Anthropic, Google---requires account registration, identity verification, and payment methods traceable to the user. When a developer queries Claude or GPT-4 for code review, the provider knows who asked, what was asked, and when. Decentralized alternatives inherit the same problem: on-chain payments link wallet addresses to service consumption, creating a permanent public record of every API call.

Buterin and Crapis~\cite{zkapi} proposed ``ZK API Usage Credits'' for anonymous inference: deposit into a smart contract, generate a zero-knowledge proof per API call to authorize payment without revealing identity. The cryptographic foundation is sound, but a Groth16 proof per API call is impractical---$\sim$640ms generation time and $\sim$300k gas on-chain per request.

We present a system where:
\begin{enumerate}[nosep]
  \item A user deposits stablecoins into a shielded pool (public, one-time).
  \item The user partially withdraws via a single ZK proof to fund a pseudonymous credit account. The remainder stays shielded as a change UTXO, unlinkable to the withdrawal.
  \item The user authorizes many job payments from the credit account via cheap ECDSA signatures ($\sim$50k gas each), with no further ZK proofs required.
  \item When the credit account is depleted, the user generates a new ZK proof for another partial withdrawal---each withdrawal is unlinkable to the previous ones.
\end{enumerate}

The operator serves requests and claims payment, but never learns the user's wallet address, deposit history, or total balance. On-chain observers see tokens flow through the shielded pool but cannot connect depositors to spenders.

\subsection{Contributions}

\begin{itemize}[nosep]
  \item A UTXO-based privacy model where partial withdrawals produce unlinkable change notes, enabling incremental spending from a single deposit without linkability across withdrawal sessions.
  \item A two-phase payment scheme: one ZK proof amortized over many ECDSA-signed job payments, reducing per-job cost from $\sim$300k gas to $\sim$50k gas.
  \item A gateway architecture bridging VAnchor shielded pools to Tangle's service lifecycle with zero modifications to audited core contracts.
  \item Multi-asset wrapping (USDC, USDT, DAI $\to$ single pool token) to maximize the anonymity set across stablecoin ecosystems.
  \item A cross-chain bridge via LayerZero V2 enabling deposits on one chain and spending on another.
\end{itemize}

\section{Preliminaries}

We adopt notation from the Webb Protocol~\cite{webb}. Let $H : \mathbb{F}_p^* \to \mathbb{F}_p$ denote Poseidon, a zkSNARK-friendly collision-resistant hash~\cite{poseidon}. Let $(\mathsf{pk}, \mathsf{sk})$ denote a keypair where $\mathsf{pk} = H(\mathsf{sk})$.

\begin{definition}[UTXO Commitment]
A commitment to a UTXO with chain identifier $i$, value $v$, public key $\mathsf{pk}$, and blinding factor $r \xleftarrow{\$} \mathbb{F}_p$ is:
$$\mathsf{cm} = H(i, v, \mathsf{pk}, r)$$
\end{definition}

\begin{definition}[Nullifier]
The nullifier for commitment $\mathsf{cm}$ at Merkle path index $j$ with signature $\sigma = H(\mathsf{sk}, j, \mathsf{cm})$ is:
$$\mathsf{sn} = H(\mathsf{cm}, j, \sigma)$$
The signature $\sigma$ binds the nullifier to the secret key, preventing nullifier grinding.
\end{definition}

\begin{definition}[Cross-Chain Merkle Membership]
Given Merkle roots $\mathbf{MR} = \{\mathsf{MR}_1, \ldots, \mathsf{MR}_k\}$ from $k$ connected anchors, commitment $\mathsf{cm}$ satisfies membership if:
$$\mathsf{reconstruct}(\mathsf{cm}, \mathsf{path}(\mathsf{cm}), H) \in \mathbf{MR}$$
Empty edge slots (unconnected chains) are filled with a deterministic zero hash.
\end{definition}

\subsection{UTXO Model and Partial Withdrawals}

The VAnchor implements a UTXO-based shielded pool with JoinSplit transactions (2-in-2-out or 16-in-2-out). Unlike fixed-denomination mixers, each UTXO carries an arbitrary value. A partial withdrawal works as follows:

\begin{enumerate}[nosep]
  \item User holds $\mathsf{UTXO}(v)$ in the pool (e.g., $v = 100$ USDC).
  \item User generates a ZK proof spending $\mathsf{UTXO}(100)$ with:
  \begin{itemize}[nosep]
    \item Output 1: withdrawal of 10 to the gateway ($\mathsf{extAmount} = -10$)
    \item Output 2: new $\mathsf{UTXO}(90)$ re-shielded in the pool (change note)
  \end{itemize}
  \item The circuit enforces $\mathsf{public\_amount} + \sum v_{\mathsf{in}} = \sum v_{\mathsf{out}}$, i.e., $-10 + 100 = 90$.
  \item The original UTXO's nullifier is marked spent. The change UTXO receives a fresh commitment with fresh blinding---\textbf{unlinkable to the original}.
\end{enumerate}

The user can repeat this process: withdraw 10 more from $\mathsf{UTXO}(90)$, producing $\mathsf{UTXO}(80)$, and so on. Each withdrawal uses fresh nullifiers and commitments. An observer sees withdrawals from the pool but cannot determine whether they originate from the same depositor.

\subsection{Shielded Gateway}

The \texttt{ShieldedGateway} is a stateless proxy. It executes a VAnchor withdrawal (validating the zkSNARK proof and marking nullifiers), receives tokens, and atomically forwards them to:

\begin{itemize}[nosep]
  \item \texttt{shieldedRequestService}: Creates a Tangle service with the gateway as owner and ephemeral addresses as permitted callers.
  \item \texttt{shieldedFundService}: Tops up an existing service's escrow.
  \item \texttt{shieldedFundCredits}: Funds a prepaid credit account (Section~\ref{sec:credits}).
\end{itemize}

\begin{property}[Atomicity]
The gateway holds no funds beyond a single transaction. If any downstream call fails, the entire transaction---including the VAnchor withdrawal---reverts.
\end{property}

\begin{property}[Non-Linkability]
The gateway stores no mapping between nullifiers and service/credit identifiers. The on-chain trace shows $\texttt{VAnchor} \to \texttt{Gateway} \to \texttt{Credits}$ but reveals nothing about which depositor initiated the flow.
\end{property}

\subsection{Shielded Credits}\label{sec:credits}

A prepaid credit system that amortizes one ZK proof over many service invocations.

\begin{definition}[Credit Account]
Identified by $\mathsf{C} = \mathsf{keccak256}(\mathsf{pk}_s \| \mathsf{salt})$ where $\mathsf{pk}_s$ is an ephemeral ECDSA public key. State: $(\mathsf{pk}_s, \mathsf{token}, \mathsf{balance}, \mathsf{nonce})$.
\end{definition}

\begin{definition}[Spend Authorization]
An EIP-712~\cite{eip712} typed signature:
$$\mathsf{SpendAuth} = \mathsf{Sign}_{\mathsf{sk}_s}(\mathsf{C}, \mathsf{serviceId}, \mathsf{jobIndex}, \mathsf{amount}, \mathsf{operator}, \mathsf{nonce}, \mathsf{expiry})$$
The designated $\mathsf{operator}$ address is the only entity that can claim the payment.
\end{definition}

The credit lifecycle: \textsc{FundCredits} pulls tokens from the gateway and initializes the account. \textsc{AuthorizeSpend} verifies the EIP-712 signature, deducts balance, increments nonce, and stores a pending claim. \textsc{ClaimPayment} transfers tokens to the designated operator. Only $\mathsf{msg.sender} = \mathsf{operator}$ can claim.

\section{Privacy Analysis}

\subsection{What the Operator Learns}

\begin{center}
\begin{tabular}{lcc}
\hline
\textbf{Information} & \textbf{Operator learns?} & \textbf{Chain observer learns?} \\
\hline
User's wallet address & No & No \\
User's deposit amount & No & No \\
User's total shielded balance & No & No \\
Which deposit funded this credit & No & No \\
Credit commitment $\mathsf{C}$ & Yes & Yes \\
Amount paid per job & Yes & Yes \\
Request content (prompt, image, etc.) & Yes & No \\
Linkage between jobs from same $\mathsf{C}$ & Yes & Yes \\
Linkage between different $\mathsf{C}$s & No & No \\
\hline
\end{tabular}
\end{center}

The operator knows \emph{what} was requested and \emph{how much} was paid, but not \emph{who} is paying. This is analogous to paying cash at a store---the merchant sees the transaction but cannot trace it to a bank account.

\subsection{Unlinkability Across Sessions}

When a credit account $\mathsf{C}_1$ is depleted, the user generates a new partial withdrawal from their shielded UTXO, producing a fresh commitment $\mathsf{C}_2$ with a new ephemeral key. The two credit accounts are cryptographically unlinkable:
\begin{itemize}[nosep]
  \item Different ephemeral keys $\mathsf{pk}_{s_1} \neq \mathsf{pk}_{s_2}$
  \item Different commitments $\mathsf{C}_1 \neq \mathsf{C}_2$
  \item Different nullifiers consumed in each withdrawal proof
  \item The change UTXO connecting them has a fresh commitment with fresh blinding
\end{itemize}

An observer who sees both credit accounts cannot determine if they belong to the same user. The anonymity set is bounded by the total number of depositors in the pool.

\subsection{Payment Security}

\begin{property}[No Double-Spend]
Balance deducted atomically before authorization is stored. Strict nonce ordering prevents replay.
\end{property}

\begin{property}[No Front-Running]
Each spend authorization names a specific operator. Only that address can call \textsc{ClaimPayment}.
\end{property}

\begin{property}[Key Immutability]
Once set, the spending key cannot be changed, preventing re-keying attacks on discovered commitments.
\end{property}

\subsection{Limitations}

\textbf{Intra-session linkability.} Multiple spends from the same credit account $\mathsf{C}$ are linkable. This is inherent to the prepaid model---full per-job unlinkability requires a ZK proof per job.

\textbf{Operator sees request content.} Payment privacy does not imply computation privacy. The operator processes the user's prompt/image/query. TEE-based execution (future work) would address this.

\textbf{Anonymity set size.} Privacy degrades if few users deposit into the pool. Multi-asset wrapping (USDC + USDT + DAI into one pool) and cross-chain deposits help grow the set.

\textbf{No IP-level privacy.} Without Tor or a mixnet, the user's network identity is visible to the operator. Orthogonal to this work.

\section{AI Service Use Cases}

The system is designed for any Blueprint-based cloud service but is primarily motivated by AI workloads where usage patterns are sensitive.

\subsection{LLM Inference}

A developer querying an LLM for code review, legal analysis, or medical information may not want the inference provider to associate those queries with their identity. With shielded payments:
\begin{itemize}[nosep]
  \item The developer deposits USDC into the pool from their known wallet.
  \item They withdraw to a credit account and submit prompts to a vLLM-based inference Blueprint.
  \item The operator serves the inference and claims payment, but cannot link the queries to the developer's wallet, ENS name, or on-chain history.
  \item When the credit is depleted, the developer creates a new credit account from their shielded balance---the new session is unlinkable to the previous one.
\end{itemize}

\subsection{Image and Video Generation}

Creative professionals generating images via Stable Diffusion or video via Sora-class models may wish to keep their prompts and creative direction private. The same flow applies: deposit once, generate many images across multiple unlinkable credit sessions.

\subsection{AI Agent Execution}

Autonomous AI agents operating on-chain need to pay for inference, tool use, and compute without revealing their controller's identity. An agent deposits into the shielded pool, funds credits, and pays for services autonomously. The agent's on-chain identity (a smart contract or EOA) is decoupled from its controller.

\subsection{Fine-Tuning and Training}

Users submitting proprietary datasets for fine-tuning can pay anonymously, preventing competitors from monitoring which models are being trained and by whom. The operator receives the data and payment but cannot attribute the training job to a specific organization.

\subsection{General Cloud Compute}

Beyond AI, any Blueprint-based service---databases, storage, CI/CD, web hosting---can accept shielded payments. The credit system is service-agnostic; it simply authorizes token transfers from a pseudonymous account to a designated operator.

\section{Cross-Chain Architecture}

The VAnchor supports cross-chain deposits via connected Merkle roots. A user deposits USDC on Ethereum, and the Merkle root is relayed to Base via LayerZero V2. The user then generates a withdrawal proof on Base, citing the Ethereum root in the cross-chain membership proof. The circuit validates membership against any of the $k$ connected roots.

The system deploys with $\mathsf{maxEdges} = 7$ (supporting 8 chains total). The circuit is compiled for 8-root membership proofs; unoccupied edge slots are filled with deterministic zero hashes. Chains are added incrementally via the \texttt{addEdge} function on the LayerZero bridge contract---no redeployment required.

Initial deployment targets: Ethereum, Arbitrum, Base, Hyperliquid, BSC.

\section{Cost Analysis}

\begin{center}
\begin{tabular}{lrrr}
\hline
\textbf{Approach} & \textbf{Per-job proof} & \textbf{Per-job gas} & \textbf{Per-job latency} \\
\hline
ZK proof per call~\cite{zkapi} & Groth16 & $\sim$300k & $\sim$640ms \\
Direct VAnchor withdrawal & Groth16 & $\sim$300k & $\sim$640ms \\
\textbf{Shielded Credits (this work)} & \textbf{ECDSA sig} & $\sim$\textbf{50k} & $<$\textbf{100ms} \\
x402~\cite{x402} (no privacy) & None & $\sim$30k & $<$50ms \\
\hline
\end{tabular}
\end{center}

The one-time ZK proof cost ($\sim$640ms, $\sim$300k gas) is amortized over all jobs in a credit session. A user who funds \$50 in credits and pays \$0.10 per inference call amortizes the proof cost over 500 calls.
